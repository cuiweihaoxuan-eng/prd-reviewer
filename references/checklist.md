# PRD 评审检查清单

## 一、文档结构完整性

| 评审项 | 状态 | 评分 | 说明 |
|--------|------|------|------|
| 产品概述 | ⬜ | / | 背景、目标、价值 |
| 用户画像 | ⬜ | / | 目标用户、角色定义 |
| 功能清单 | ⬜ | / | 功能列表及优先级 |
| 业务流程 | ⬜ | / | 流程图、活动图 |
| 页面结构 | ⬜ | / | 信息架构、页面清单 |
| 详细需求 | ⬜ | / | 功能详细描述 |

| API接口 | ⬜ | / | 接口清单、协议 |
| 非功能需求 | ⬜ | / | 性能、安全、兼容性 |
| 验收标准 | ⬜ | / | 可测试的验收条件 |

## 二、需求描述质量

### 2.1 清晰性
- [ ] 每个功能有明确的业务目标
- [ ] 描述使用"用户可以..."句式
- [ ] 包含前置条件（什么情况下触发）
- [ ] 包含后置条件（操作后的结果）
- [ ] 无歧义表述（如"适当的"、"合理的"）

### 2.2 完整性
- [ ] 覆盖正向流程
- [ ] 覆盖异常/边界情况
- [ ] 包含权限控制描述
- [ ] 包含数据校验规则

### 2.3 可测试性
- [ ] 每个需求有明确的验收标准
- [ ] 验收标准可量化（如"响应时间<200ms"）
- [ ] 可通过功能测试验证

### 2.4 一致性
- [ ] 功能间无逻辑冲突
- [ ] 与现有系统无冲突
- [ ] 术语定义统一

## 三、技术方案评审

### 3.1 架构合理性
- [ ] 技术选型是否合适
- [ ] 模块划分是否清晰
- [ ] 扩展性考虑

### 3.2 数据库设计
- [ ] ER图完整
- [ ] 索引设计合理
- [ ] 数据类型选择合适

### 3.3 API设计
- [ ] RESTful规范
- [ ] 错误码覆盖
- [ ] 版本管理

## 四、页面设计评审（如有原型）

| 评审维度 | 评审要点 |
|----------|----------|
| 布局合理性 | 视觉层级，信息分组、留白 |
| 交互一致性 | 按钮位置、反馈机制、导航逻辑 |
| 组件规范性 | 按钮、表单、列表、弹窗样式统一 |
| 无障碍设计 | 颜色对比、焦点顺序、替代文本 |
| 响应式适配 | 移动端/平板/桌面布局 |
| 用户引导 | 空状态、新手引导、错误提示 |

## 五、非功能需求评审

| 需求类型 | 评审指标 |
|----------|----------|
| 性能 | 响应时间、并发数、吞吐量 |
| 可用性 | uptime、故障恢复 |
| 安全性 | 权限控制、数据加密、审计日志 |
| 兼容性 | 浏览器、设备、操作系统 |

## 六、评分标准

| 评分 | 说明 |
|------|------|
| A | 优秀，完全满足要求 |
| B | 良好，基本满足要求，有小问题 |
| C | 一般，部分不满足要求，需要改进 |
| D | 较差，多处不满足要求，需要重大改进 |

## 七、问题严重程度

| 严重程度 | 说明 |
|----------|------|
| 🔴 高 | 必须修复，否则无法交付 |
| 🟡 中 | 建议修复，影响用户体验 |
| 🟢 低 | 可选优化，不影响功能 |
