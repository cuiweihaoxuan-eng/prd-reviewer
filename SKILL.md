# PRD 评审工具

本技能用于评审PRD（产品需求文档）文档，提供结构化的评审维度和输出模板。

## 何时使用

- 用户要求评审PRD文档
- 用户要求评审技术方案设计文档
- 用户要求评审功能模块设计
- 用户要求评审原型设计（Figma、蓝湖、在线原型链接）
- 用户提到"评审"、"评审PRD"、"需求评审"、"方案评审"、"原型评审"等关键词

## 使用流程

### 1. 读取PRD文档

使用Read工具读取PRD文档内容，了解文档结构和内容。

### 2. 自动识别PRD类型

分析文档特征，自动识别PRD类型：

| 特征 | 整体PRD | 独立功能模块 | 现有功能新增模块 |
|------|---------|--------------|------------------|
| 技术架构章节 | ✅ 有 | ❌ 无 | ❌ 无 |
| 项目结构说明 | ✅ 有 | ⚠️ 简略 | ⚠️ 引用现有 |
| 系统集成说明 | 独立系统 | 新增模块 | 需对接现有系统 |

**识别规则**：
- 有完整技术架构 → 整体PRD
- 有简略项目结构，无架构 → 独立功能模块
- 提及"现有系统"、"集成" → 新增模块

### 3. 确定评审范围

根据PRD类型确定评审范围：

| PRD类型 | 评审范围 |
|---------|---------|
| 整体PRD | 完整评审所有维度 |
| 独立功能模块 | 需求描述 + 技术方案（简化版） |
| 现有功能新增模块 | 新增功能需求 + 集成 + 数据兼容 |

### 4. 执行评审

使用references/prd评审清单执行评审：

#### 文档结构评审
- 产品概述、用户画像、功能清单
- 业务流程、数据模型、API接口
- 非功能需求、验收标准

#### 需求描述评审
- 业务目标明确
- 描述清晰（前置/后置条件）
- 完整性（正向+异常流程）
- 可测试性（量化指标）

#### 技术方案评审
- 技术选型合理性
- 数据库设计
- API设计
- 扩展性

### 5. 输出评审报告

按照以下JSON格式输出评审结果：

  "评审配置": {
    "评审范围": {

      "需求质量": "✅",

    }
  },
  "评审结果": {
    "整体评分": "B",
    "严重问题": [],
    "建议改进": [],
    "优点": []
  },
  "详细评审": {
    "文档结构": { "状态": "✅ 评审", "评分": "B", "问题": [] },

## 评审状态标记

| 标记 | 含义 |
|------|------|
| ✅ 评审 | 本次需要评审 |
| ❌ 不适用 | 功能不涉及此项 |
| ⏭️ 不评审 | 本次暂不评审 |

## 评审维度说明

### 文档结构
- 是否包含产品概述、用户画像
- 是否有功能清单（带优先级）
- 是否有业务流程图
- 是否有验收标准

### 需求质量
- 业务目标是否明确
- 是否有前置/后置条件
- 是否覆盖异常流程
- 是否有量化指标

### 技术方案
- 技术选型是否合理
- 数据库设计是否完整
- API设计是否规范

### 页面设计（如有原型）
- 布局合理性
- 交互一致性
- 组件规范性

### 页面原型评审（如有Figma/在线原型链接）

当用户提供Figma原型链接或在线原型地址时，执行以下评审：

#### 评审方式

1. **在线原型链接**：使用 agent-browser 技能打开原型链接，浏览各个页面并截图
2. **Figma链接**：解析Figma原型链接或使用浏览器访问
3. **截图分析**：使用 Read 工具分析截图内容

#### 评审维度

| 评审维度 | 评审要点 |
|----------|----------|
| 布局合理性 | 视觉层级、信息分组、留白、响应式适配 |
| 交互一致性 | 按钮位置、反馈机制、导航逻辑、状态切换 |
| 组件规范性 | 按钮、表单、列表、弹窗、输入框样式统一 |
| 信息架构 | 页面层级、导航结构、内容组织 |
| 用户体验 | 空状态、加载状态、错误提示、引导流程 |

#### 输出格式

```json
{
  "原型评审": {
    "原型类型": "Figma原型/在线原型/设计稿",
    "原型地址": "xxx",
    "评审结果": {
      "整体评分": "B",
      "布局合理性": { "评分": "B+", "问题": [] },
      "交互一致性": { "评分": "B", "问题": [] },
      "组件规范性": { "评分": "A", "问题": [] },
      "用户体验": { "评分": "B-", "问题": [] }
    }
  }
}
```

## 常见问题处理

### 1. PRD类型判断模糊
如果无法明确判断PRD类型，优先按"独立功能模块"处理，只评审需求描述和技术方案。

### 2. 缺少原型截图
如PRD中无原型截图或链接，页面设计部分标记为"❌ 不适用"。

### 3. 增量评审
对于现有系统新增模块，评审重点放在：
- 新增功能需求
- 与现有系统的集成点
- 数据兼容性

### 4. 原型链接评审
当用户提供Figma原型链接或在线原型地址时：
1. 使用 agent-browser 技能打开原型链接进行浏览
2. 对关键页面进行截图
3. 使用 Read 工具分析截图中的UI设计
4. 输出原型评审结果

## 输出要求

1. 首先输出PRD类型识别结果和评审范围配置
2. 然后按维度输出详细评审
3. 最后输出改进建议
4. 保持评审结果客观、结构化
